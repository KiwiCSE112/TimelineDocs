1) Space Shuttle

• Shuttle process is flawed
• Shuttle process is good
• Why Article describes a Waterfall approach - and is incorrect for today’s requirements.
• Crucial software requires this approach. Ensuring minimal errors is the way to go.
• Shifting time spent in a bug fix release but instead just moving it before the release.
• If it affects a small number of people and the bugs are really small, it’s OK. Efficiency
• Time to use the “Space Shuttle Approach” should be used when you have the time and money to do so.
• A small startup doesn’t have these kinds of resources.
• An organization as big as NASA can support the time and resources required. It is not practical for smaller organizations.
• Process can be applied to everything because they have a lot of important things in the process
• No cutting corners
• Follow process even if you don’t have all the specs

• Upset shareholders - they’ll be upset you take so long for the first deliverable.
• Market might have ‘moved’ on.
• Fixed work schedule

• Projects need to be dynamic - customers and market change. Having thousands of pages of documents isn’t practical especially knowing there will be plenty of changes.
• Little incentive to write code like NASA
• Responsibility for safety of users, follow process
• Be more like software engineers than coders

• DIfferent processes work for different teams. This process works especially well for NASA. Majority of software out there is primarily customer driven. If you can make modifications in time, you won’t be able to sell your product.
• Documentation is excessive for majority of software engineering, however, it shows a good example that documentation is very important because it makes you think about every line of code you write
• The overall process is good because it encourages thinking about your code before writing it, leading to fewer bugs

2) Decide React or Web Components

Web Components

Pros
• Powell Recommended
• Good for long term projects
• Polymer <3 provides the compatibility
Cons
• Longer time to implement
• High learning curve

React v16.3

Pros
• Performance testing is easy
• Lots of previous experience with web components
• We can fail fast
Cons
• Slower than Web Components
• SEO is a bit of a problem

Consensus:

We will use React for the next weeks or so. If we find that it doesn’t satisfy our needs, then we will rapidly switch to Web Components.
